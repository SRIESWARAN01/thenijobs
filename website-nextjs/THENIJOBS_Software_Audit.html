Executive Summary

THENIJOBS is a well-organised, modern, single-codebase platform that already does many things right — strong Firestore/Storage security rules, server-validated job posting, scheduled lifecycle automation, a coherent design system, and clean TypeScript throughout. It is not a beginner project. The gaps that matter are concentrated in four areas: (1) a static-export architecture that blinds the product to SEO, (2) a self-assigned role model that exposes job-seeker data, (3) a manual, non-gateway payment flow that throttles revenue, and (4) silent cross-user write failures caused by a mismatch between client code and security rules.

1 · Project Structure & Architecture

1.1 What the codebase is

A single Next.js 16 App-Router project (package.json → next 16.2.7, react 19.2.4) compiled as a fully static export (next.config.ts → output: 'export', images.unoptimized: true). All dynamic data is read at runtime on the client directly from Firebase. There is no Next.js server, no API routes, and no middleware. Server-side logic lives in two places only: Firestore/Storage security rules and a small set of Firebase Cloud Functions (functions/src/index.ts).

1.3 Architecture-level findings

A1 — Static export defeats SEO for the most valuable pages. HIGH
With output:'export', dynamic routes cannot be server-rendered. The team worked around Next's static-param requirement by shipping two parallel route trees: real param routes (app/jobs/[id]/page.tsx, app/company/[slug]/page.tsx, app/profile/[uid], app/id/[uid]) and query-string "QueryClient" variants (app/jobs/detail/JobDetailQueryClient.tsx, app/company/CompanyProfileQueryClient.tsx, app/profile/PublicProfileQueryClient.tsx, app/id/DigitalIdQueryClient.tsx). Job and company detail content is fetched client-side after load, so Googlebot sees an empty shell. For a local-jobs/business-directory product whose entire growth model is organic search ("jobs in Theni"), this is a direct revenue limiter despite the strong metadata in app/layout.tsx and the presence of sitemap.ts/robots.ts.
A2 — Hosting config contradicts the build target. MEDIUM
firebase.json declares hosting.frameworksBackend.region (an SSR construct) while next.config.ts emits a static export. The frameworks backend is effectively dead config. Decide on one model: either drop output:'export' and use the Firebase web framework SSR (fixes A1), or keep static export and remove the misleading backend config.
A3 — Route protection is client-only. MEDIUM
Because static export forbids middleware, every guard runs in the browser via useRequireAuth() (src/hooks/useAuth.ts) which redirects in a useEffect. Protected pages therefore mount and may briefly render before redirecting. Data is still safe (Firestore rules are the real boundary), but it is a UX/perception issue and means SEO/preview crawlers can see protected page scaffolding.

1.4 Dead / duplicate / unused

  • Dead branch: functions/src/index.ts:313 — status: plan === 'free' ? 'active' : 'active' (both branches identical; a no-op ternary that signals an unfinished status rule).
  • Duplicate route trees: the [param] vs /detail?… pairs described in A1 are maintenance duplication.
  • Dependency hygiene: lucide-react is pinned to ^1.17.0 in package.json, which does not correspond to lucide-react's published 0.x release line — verify the lockfile resolves to a real version to avoid a surprise install failure in CI.
  • Bleeding-edge core: Next 16 / React 19 are very new; pin exact versions and watch for breaking patches before launch.
  • No tests: there is no test runner or test files; npm run verify only lints, typechecks and builds. For a payments-adjacent product this is a maintainability gap.

4 · Workflow Analysis

4.1 Registration → verification

Current (app/register/page.tsx → AuthContext.createAccount): choose role → enter name/email/phone/password → createUserWithEmailAndPassword → write users/{uid} → seed a free yearly subscriptions/{uid}_free doc → seed seekerProfiles + publicProfiles (seekers) → sendEmailVerification → immediately sign the user out with a "verify your email then sign in" message.

Problems / missing steps: role is fully self-asserted (any visitor can pick "employer"); password floor is 6 chars; phone is collected but never verified (no OTP); the forced sign-out-after-signup is unusual UX (most users expect to land in the app). Recommended: keep email verification, but consider verifying inside the session (banner + gated actions) instead of an immediate logout; add phone OTP if phone is used for contact/trust; gate "employer/business" privileges behind verification or admin review rather than self-selection.

4.2 Login → role redirect

Current (app/login/page.tsx → signInWithEmail → onAuthStateChanged): on sign-in, rejectUnverifiedEmail reloads the user, and if unverified, re-sends verification and signs them back out. Verified users get their profile fetched and are routed by getSafePostLoginRedirect() (src/lib/access.ts) which safely whitelists same-portal redirect paths and otherwise sends to the role dashboard. Assessment: solid; the redirect allow-listing is a good open-redirect defence.

4.3 Job posting (the model workflow)

Current (employer/post-job → firestoreService.createJobPosting → callable createJobPosting in functions/src/index.ts): the function enforces email-verified, role ∈ {employer, business_owner}, company ownership, company-not-deleted, plan job limits, feature gating (featured/urgent/premium), duplicate detection, and spam flags, then writes the job as status:'pending', isActive:false. Admin approval (approveJob) flips it live and stamps a 30-day expiry. Assessment: this is the strongest workflow in the system — server-authoritative, well-guarded. Minor: approveJob re-reads all company jobs + subscriptions client-side to re-check limits (works, but unbounded).

4.4 Job application

Current (firestoreService.applyToJob): verifies the job is publicly visible, writes applications/{seekerId_jobId} (deterministic ID blocks duplicates), increments the job counter, logs activity, and tries to notify the employer. Problem (HIGH): the employer notification is created client-side with userId = company.ownerId, but the Firestore rule for notifications only permits create when request.resource.data.userId == request.auth.uid or admin. A seeker is neither → the write is denied and silently swallowed by runSideEffect. Employers therefore may never get "new application" alerts. (See §9 for the fix — route these notifications through a Cloud Function.)

4.5 Premium purchase / payment / subscription

Current: there is no payment gateway. Pricing CTAs call firestoreService.createPaymentRequest which writes a paymentRequests doc with status:'pending'. Firestore rules let users create only free subscriptions; paid subscriptions/payments can be created only by an admin (firestore.rules lines 272–292). So upgrades require a human admin to receive payment out-of-band, then manually create the subscription. The scheduled function then manages expiry/renewal reminders and syncs isPremium back onto the company/seeker. Problem (HIGH): manual collection throttles revenue, adds latency between intent and access, and is unscalable. Recommended: integrate Razorpay/Cashfree (India, UPI) via a Cloud Function that verifies the payment signature server-side and writes the subscription — preserving the "only server creates paid subs" invariant the rules already assume.

4.6 Notifications

Current: dual model — Cloud Functions create notifications via Admin SDK (works: subscription/job expiry) while client code also tries to create cross-user notifications (fails, §4.4). Recommended: make a single createNotification callable the only writer of notifications for another user.

4.7 Admin moderation

Current (firestoreService.approve/reject Company/Job, verifyUser, updateUserRole): admins act through direct Firestore writes, gated by isAdmin() in rules. Activity is logged to activityLogs. Assessment: functional and audited; consider moving privilege-changing actions (role changes, verification) to callable functions for centralised validation and custom-claim syncing.

On the "auto-logout / session" concern specifically: the implementation already keeps users signed in indefinitely until they log out — persistence is browserLocalPersistence and tokens auto-refresh. The only involuntary sign-outs are (a) unverified email and (b) explicit logout. If users report being logged out, the likely culprit is the unverified-email re-check on onAuthStateChanged (AuthContext.tsx lines 219–232, 240–245), which calls reload() + getIdToken(true) and signs out anyone still unverified. Confirm users have verified before assuming a session bug.

5.1 Auth fixes (concise)

  • Replace forced post-signup logout with an in-session "verify to continue" gate.
  • Raise password minimum to 8 and add a strength indicator on register.
  • When an admin sets a role (updateUserRole), also set the matching custom claim via a callable so rules and claims never diverge.
  • Map raw Firebase error codes to friendly copy in login/register (e.g. auth/invalid-credential → "Email or password is incorrect").

6 · Performance Audit

6.1 What's already good

  • React Query (@tanstack/react-query) for client cache/dedupe.
  • Counts use getCountFromServer (cheap aggregation) in getPlatformStats/getEmployerStats/getSeekerStats.
  • Independent reads are parallelised with Promise.all.
  • Immutable long-cache headers for static assets (firebase.json: max-age=31536000, immutable).
  • Composite indexes are defined for the hot queries (firestore.indexes.json).

6.3 Optimization strategy

  1. Add limit() + cursor pagination to every public list (start at 20–24).
  2. Collapse getPublicCompanies to a single indexed query; migrate legacy status:'approved' docs to verificationStatus:'verified' once, then drop the second read.
  3. writeBatch for bulk notification reads-as-read.
  4. Adopt SSR/ISR (drop output:'export') so detail pages stream server-rendered HTML — fixes P5/P7 and SEO together.
  5. Generate resized Storage thumbnails (Firebase resize extension) and serve those in cards.
  6. Wrap decorative animations in @media (prefers-reduced-motion: no-preference).

7 · Security Audit

8 · Database Audit

Firestore is used as the system of record. The model (in types/index.ts) is comprehensive: users, seekerProfiles, publicProfiles, companies, jobs, applications, savedJobs, reviews, leads, interviews, notifications, jobAlerts, subscriptions, payments, paymentRequests, services, serviceRequests, advertisements, conversations/messages, activityLogs, supportTickets, franchises, broadcasts, settings.

8.1 Strengths

  • Deterministic IDs prevent duplicates where it matters: applications/{seekerId_jobId}, savedJobs/{userId_jobId}, subscriptions/{uid}_free.
  • Denormalised company fields onto jobs (name, slug, logo, status) avoid per-job company joins on list pages — appropriate for Firestore.
  • Composite indexes cover the main filter+sort combinations (firestore.indexes.json).

11 · Business Logic Audit

The model is a single-district (Theni, Tamil Nadu — types/index.ts LAUNCH_DISTRICT='Theni') multi-sided marketplace: job seekers, employers, business owners, suppliers, service providers, entrepreneurs, plus admin/franchise roles. Monetisation is yearly subscriptions: Free ₹0 / Basic ₹480 / Premium ₹1,200 (lib/subscriptions.ts), with feature/limit gating enforced server-side for job posting.

11.1 Strengths

15.1 E-02 · Employer notifications fail silently

Problem. Seekers create notifications docs for the employer; the rule only allows userId == auth.uid or admin → denied; runSideEffect hides it.

Why it matters. Employers never learn about applications; the marketplace looks dead.

Files. src/lib/firebase/firestoreService.ts (applyToJob, createNotification), functions/src/index.ts, firestore.rules (notifications).

Fix — add a callable, call it from the client:

// functions/src/index.ts
export const createNotification = onCall({ region: REGION, enforceAppCheck: true },
  async (req) => {
    const uid = requireUid(req);
    const { userId, type, title, message, actionUrl } = req.data ?? {};
    if (!userId || !title) throw new HttpsError('invalid-argument', 'userId and title required');
    // optional: verify the caller is allowed to notify userId (e.g. shares an application/job)
    await db.collection('notifications').add({
      userId, type: getString(type,'system'), title: getString(title),
      message: getString(message), actionUrl: getString(actionUrl),
      read:false, isRead:false, createdAt: FieldValue.serverTimestamp(),
    });
    return { ok: true };
  });
// firestoreService.ts — replace direct cross-user writes
const notify = httpsCallable(functions, 'createNotification');
await notify({ userId: company.ownerId, type:'application_update',
  title:'New Job Application', message:`${data.seekerName} applied to ${data.jobTitle}`,
  actionUrl:'/employer/candidates' });

Outcome. Employers reliably receive application alerts; self-notifications (own userId) can stay as direct writes.

15.2 E-01 · Stop self-role bulk PII access

Problem. firestore.rules 344–345 lets any hasEmployerRole() user read every seeker profile, and roles are self-set at signup.

Fix (rules). Require a trust flag the user can't self-set:

// firestore.rules — publicProfiles read
allow read: if isAdmin()
  || (isAuthenticated() && resource.data.ownerId == request.auth.uid)
  || (resource.data.type == 'job_seeker'
       && hasEmployerRole()
       && get(/databases/$(database)/documents/users/$(request.auth.uid)).data.employerVerified == true)
  || resource.data.type in ['employer','business_owner'];

Set employerVerified only via an admin action / paid plan (server-side). Also trim buildPublicSeekerProfile to non-identifying fields for list/search and reveal contact only after an application exists.
